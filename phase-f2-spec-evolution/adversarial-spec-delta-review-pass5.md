ADV-F2-P5-001 (CRITICAL) — Fail-loud "never silently discarded" guarantee not enforced at the deterministic layer. BC-3.03.001 v1.13 emitter (Inv#4 STEP 3→STEP 5) + architecture-delta §D-DEC-008 emitter pseudocode: a review marker is emitted ONLY when action == create-review/comment-review (STEP 3). A hard-floor verdict arriving with ticket_action_type ∈ {comment,create,assign,none} falls to STEP 5 (IF action==none OR hard_floor_applies(): emit allow without marker; RETURN) — no branch emits a review marker or writes an error artifact → finding silently dropped (no marker, no Jira surface, no error). Contradicts BC-10.01.001 v1.9 Inv#10 ("disposition-guard issues restricted review markers for hard-floor categories regardless of SKILL.md behavior, never silently discarding") and architecture's dual-enforcement claim; the architecture's own Fail-loud text concedes a silent-discard "is a bug in the monitoring-loop SKILL.md" — i.e. only correct LLM behavior prevents it, the exact untrusted layer the marker design defends against. VP-HOOK-029 is scoped to "hard-floor verdict WITH ticket_action_type ∈ {create-review,comment-review}" so it CANNOT detect the silent-discard case (action=comment/none); mutant SM-32 likewise undetectable. Brief §3.3/§3.9 make Indeterminate/hard-floor surfacing a hard floor; a single LLM mislabel → clean-clearance silent failure (§3.7 could-not-observe ≠ nothing-found). FIX: disposition-guard already computes hard_floor_applies(); in STEP 5 when hard_floor_applies() true (or disposition==Indeterminate) AND action not a review type, the hook MUST either (a) upgrade to the corresponding review marker (create-review if ticket_id null else comment-review) or (b) write an explicit error artifact and deny. Re-scope VP-HOOK-029 to inject hard-floor verdicts with NON-review ticket_action_type and assert (review-marker XOR error), making SM-32 killable.

ADV-F2-P5-002 (MAJOR, borderline CRITICAL) — create-review/comment-review exemption unconditional on the LLM-supplied action token, defeating the autonomy_enabled kill switch and deviating from brief §3.9. BC-3.03.001 v1.13 emitter STEP 3 (+ architecture §D-DEC-008) runs IF action==create-review → WRITE_MARKER BEFORE STEP 4 (kill switch) and STEP 5 (hard_floor_applies), with NO precondition that the verdict is actually hard-floor/Indeterminate. Any verdict (incl disposition=TP, non-hard-floor, autonomy_enabled=false) that sets ticket_action_type=create-review gets a marker exempt from kill switch + hard-floor. Consequence: autonomy_enabled=false ("emergency circuit-breaker", brief §3.9) is bypassable for jr issue create --project <key> and jr issue comment <ticket_id> by labeling the action create-review/comment-review. Verdict construction is LLM-driven and the marker threat model (D-DEC-001/SEC-001) assumes the LLM surface is injectable → an attacker steering the loop executes Jira writes while the operator believes it halted. Brief §3.9 says the kill switch preserves "evidence collection and Jira DRAFTING" — authorizes drafting, not ticket creation/commenting via jr; D-DEC-012 executes jr create/comment under autonomy_enabled=false = spec-vs-brief conflict. The "restricted" create-review command_pattern is identical to regular create (^jr (--output json )?issue create --project <key>( |$)) with no --label/--summary constraint (architecture rejects label-content enforcement, pushes [REVIEW-REQUIRED]/[BLIND-SPOT] to a SKILL.md Iron Law) → a kill-switch-exempt create-review marker authorizes an effectively arbitrary jr issue create. FIX: gate STEP 3 on the deterministic hard_floor_applies(verdict) OR disposition==Indeterminate check the hook already has — refuse review-marker exemption for a verdict that is not genuinely hard-floor. Reconcile with brief §3.9 (draft-only under kill switch, or amend the brief's "Jira drafting" language with recorded human sign-off).
