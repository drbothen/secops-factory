## Description

The monitoring-loop is the secops-factory's autonomous patrol mode. It runs
unattended on a schedule (cron or Task Scheduler) and performs full Tier-1 alert
triage using a 9-stage pipeline (Stage 0 per-org/sensor health check, then 8
per-alert stages). For each registered org, it ingests new events since the last
watermark, validates and categorizes them, enriches via prism ThreatIntel/NVD
and web sources (Perplexity + Tavily), scores by asset criticality and rule
fidelity, produces one of four canonical dispositions (TP/FP/BTP/Indeterminate),
executes a Jira-first ticket action, and persists an ICD-203-grade defensible
verdict record. Hard floors unconditionally prevent autonomous action on
high-severity, critical-asset, and Indeterminate categories regardless of
configuration. All Jira write operations require a disposition-guard-issued marker
(D-DEC-001).

## Preconditions

1. The loop is invoked by the cron wrapper script `skills/monitoring-loop/run-monitoring-loop.sh`
   (or `.ps1`) as:
   ```
   claude -p "/monitoring-loop" \
     --strict-mcp-config \
     --mcp-config ~/.claude/prism.mcp.json \
     --allowedTools "mcp__prism__*,mcp__tavily__tavily_search,mcp__tavily__tavily_extract,mcp__perplexity__perplexity_ask,mcp__perplexity__perplexity_search,Bash,Write,Read,Edit" \
     --output-format json \
     < /dev/null
   ```
   (D-DEC-003, D-DEC-010). Confidence: architecture-delta.md §D-DEC-003.
2. The `--bare` flag is NOT present in the invocation. `--bare` disables hook
   enforcement (require-review, disposition-guard) and is explicitly prohibited.
   A cron wrapper containing `--bare` is a defect. (D-DEC-003). Confidence:
   architecture-delta.md §D-DEC-003 "bare explicitly forbidden."
3. `${CLAUDE_PLUGIN_DATA}` is set or defaults to the OS-appropriate path:
   Linux: `~/.local/share/claude`; macOS: `~/Library/Application Support/Claude`;
   Windows: `%APPDATA%\Claude`. The path must be writable by the OS scheduler
   process. (D-DEC-002). Confidence: architecture-delta.md §D-DEC-002.
4. `~/.claude/prism.mcp.json` exists and is valid JSON (written by activate skill,
   D-DEC-003). The cron wrapper validates its presence at startup and exits 1 if
   absent. Confidence: architecture-delta.md §D-DEC-003 wrapper script.
5. `prism.toml` is accessible at the configured spec_dir and contains at least
   one `[[orgs]]` entry with `org_slug` and `uuid` fields. Confidence: inferred
   from brief §3.2 "FOR EACH customer (org_slug) in prism.toml [[orgs]]."
6. `RUST_LOG=off` is present in the `prism` MCP server env block in
   `~/.claude/prism.mcp.json`. If omitted, prism's tracing subscriber writes
   structured JSON to stdout, corrupting MCP JSON-RPC framing and causing all
   prism tool calls to fail with parse errors. (ASM-002). Confidence:
   brief §2.1 "RUST_LOG=off must always be injected."
7. `jr` is authenticated and the Jira project key is configured (validated at
   activate time, brief §4.4). Confidence: inferred from Jira ticket action
   requirements in Stage 7.
8. **[NEW v1.6] Stage-7 verdict file path MUST contain the substring `"verdict"`.** The
   disposition-guard hook fast-path-allows any Write event whose `file_path` does not
   contain either `"investigation"` or `"verdict"` as a substring (BC-3.03.001 PC#1).
   If the monitoring-loop writes its Stage-7 verdict record to a path that lacks the
   `"verdict"` substring, the disposition-guard fast-path fires — ICD-203 validation
   is skipped, no marker is written, and every Stage-8 `jr issue` call is
   unconditionally denied by require-review (no marker to consume). The canonical

   any path matching `*verdict*` is compliant. The cron wrapper's Stage-7 Write call
   MUST use a path containing `"verdict"` as a substring. (ADV-F2-P3-005 MEDIUM.)

## Invariants

1. **org_slug scoping on ALL PrismQL queries.** Every PrismQL query issued by
   the monitoring loop MUST include an explicit `org_slug` scope constraint
   (D-DEC-005). The loop must never construct a query without this constraint.
   This is the sole plugin-side cross-tenant isolation guarantee (prism enforces
   its own RBAC; the plugin must not bypass it by querying without scope).

   **Verification property (ADV-F2-008, FINALIZED v1.3):** VP-SKILL-059 (scan-threats org_slug scoping — Iron Law content assertion) provides partial coverage for sibling skills. **VP-SKILL-064 — monitoring-loop PrismQL never lacks org_slug WHERE clause** is FINALIZED (verification-delta.md v1.1 §7 Part B): sole plugin-side cross-tenant isolation guarantee; strategy = prism-DTU multi-org fixtures asserting org-a query returns zero org-b/c rows + static org_slug-injection check + unscoped-query-rejected adversarial fixture.
2. **Indeterminate verdicts NEVER auto-closed.** This is a hard floor — an
   unconditional code branch in the monitoring-loop SKILL.md, NOT a
   configuration-evaluated condition. No value of `autonomy_enabled`, `require_review`,
   or any other config option can cause an Indeterminate verdict to be auto-closed
   or auto-resolved. Indeterminate always produces: (a) human-review-required
   Jira ticket (`[REVIEW-REQUIRED]` label) or draft, and (b) `requires-human-approval`
   annotation. Violating this invariant is a critical safety regression. D-DEC-008,
   brief §3.3, [Q1, Q6].
3. **Never auto-reopen a Closed Jira ticket.** When the loop finds a Closed
   ticket with the same root cause as a current alert, it ALWAYS opens a NEW
   ticket and links it to the closed one. This is unconditional — no configuration
   option enables auto-reopen of Closed tickets. (brief §3.4 rule #4, [Q3
   citing [27]]). Resolved tickets are different: the loop proposes a reopen but
   does NOT execute it without human approval (propose-only).
4. **Watermark monotonicity.** The watermark for each org × sensor pair is never
   set to a timestamp earlier than the previous persisted value. A future-dated
   watermark (adversarial or clock-skew signal) is rejected on read and the 24h
   fallback is used instead (D-DEC-002). Confidence: architecture-delta.md
   §D-DEC-002 READ_WATERMARK validation.
5. **Sensor silence is a positive risk finding, not null.** If a sensor's
   `last_seen_ts < now() - INTERVAL 24h` AND `row_count == 0`, the loop emits a
   BLIND-SPOT finding with `disposition = Indeterminate-due-to-missing-telemetry`.

   > **Previous (v1.7):** `last_seen_ts > now() - INTERVAL 24h` — this condition was INVERTED. It evaluated to true when the sensor WAS seen within the last 24h (i.e., NOT silent), meaning the BLIND-SPOT path fired for active sensors and was never triggered for truly silent ones. (ADV-F2-P4-003 MAJOR fix.) The corrected `<` condition fires when `last_seen_ts` is MORE than 24h ago (sensor has been silent for >24h). Verification: canonical test vector EC-006 "last_seen_ts = 36h ago" — `(now - 36h) < (now - 24h)` = true → BLIND-SPOT correctly fired.
   The loop MUST NOT emit "nothing to report" for a silent sensor. This conflates
   "no threat found" with "could not observe" — a silent failure mode that a
   consuming LLM agent would misread as a clean clearance. (brief §3.7, [Q5]).
6. **sensor_health_status field in every verdict.** Every verdict record must
   include `sensor_health_status` with value exactly one of: `healthy` (sensor
   reported within threshold, data complete), `degraded` (partial data or parse
   errors), `silent` (no data; verdict is Indeterminate-due-to-missing-telemetry).
   A `silent` or `degraded` status QUALIFIES the verdict; it does not suppress
   the verdict. (brief §3.7).
7. **Known-FP/dedup check before enrichment spend.** Stage 2 (validate + known-FP
   lookup) is executed before Stage 4 (enrich). An event that exits at Stage 2
   (known-FP match → FP disposition, skip enrichment) must never proceed to
   enrichment. This ordering reduces enrichment API costs by 30–40% in
   well-tuned environments ([Q1 §"known-pattern lookup first"]).

   **ADV-F2-005 clarification (v1.2):** This invariant applies ONLY to the enrichment spend. **ADV-F2-P2-002 correction (v1.4):** The enrichment spend that is skipped is Stages 4–5 (ENRICH and SCORE). Stage 3 (CATEGORIZE — MITRE ATT&CK mapping) still executes on the known-FP fast-exit path to populate `verdict.attack_techniques` for hard-floor evaluation. Hard-floor evaluation (Invariant #10) is NOT bypassed by a known-FP Stage-2 match — hard floors are evaluated BEFORE any auto-close action regardless of known-FP match. See EC-009 for the updated fast-exit behavior.
8. **BLIND-SPOT ticket dedup (D-DEC-004).** At most one open BLIND-SPOT ticket
   per `(org_slug, sensor_id)` pair may exist at any time. Before creating a new
   BLIND-SPOT ticket, the loop executes the dedup JQL query. If an open ticket
   exists, the loop appends a comment. If a Closed ticket exists (same org/sensor),
   the loop creates a NEW ticket and links it — never auto-reopens.

   **Verification property (FINALIZED v1.5):** VP-SKILL-068 — in-grace re-fetched event with existing open ticket → COMMENT not new ticket; dedup JQL finds open (org_slug, sensor_id) ticket → append-comment action, no new ticket created (verification-delta.md v1.2 §7 Part C).
9. **[UPDATED v1.2] All 15 mandatory fields present in every verdict record.** The
   disposition-guard hook (BC-3.03.001 v1.13) blocks ticket action if any of
   the following are absent (null or missing key — empty array is allowed for
   list fields, but the key must be present):
   ```
   1.  disposition           (TP|FP|BTP|Indeterminate — never free-text)
   2.  confidence            (ENUM-ONLY: high|medium|low — derived from confidence_score float via D-DEC-011
                              canonical thresholds: high ≥ 0.75, medium ≥ 0.40 and < 0.75, low < 0.40;
                              disposition-guard validates this field as an enum, NOT a float — a float
                              confidence value (e.g. 0.85) is DENIED by VP-HOOK-025 float-reject assertion
                              (per-class split: investigation-markdown 12-field path / verdict-JSON 15-field
                              path + enum type assertion; verification-delta.md v1.3 §2 / ADV-F2-P3-008);
                              the float posterior is stored separately as confidence_score — see
                              Non-ICD-203 operational metadata fields note below)
   3.  sensor_health_status  (healthy|degraded|silent)
   4.  evidence_artifacts    (array of {source, fetch_time_utc, ocsf_normalized_id, hash})
   5.  timeline_events       (array of {ts_utc, event_type, actor_type(agent|human), description})
   6.  hypotheses_considered (array of {hypothesis, supporting_evidence, contradicting_evidence, verdict})
   7.  alternatives_rejected (array of {alternative, reason_rejected})
   8.  uncertainty_explicit  (string or null — null only if no uncertainty present)
   9.  attack_techniques     (array of {mitre_technique_id, mitre_tactic, confidence})
   10. agent_actions         (array of {ts_utc, action, result, requires_human_approval: bool})
   11. human_actions         (array of {ts_utc, actor_id, action})
   12. tuning_signal         ({rule_id, asset, reason} or null — null for TP/Indeterminate; required non-null for FP/BTP)
   13. severity              (LOW|MEDIUM|HIGH|CRITICAL — matches alert.severity from sensor data at Stage 5;
   ```
